Slide 3 of 5 — Data Cleaning + Model Layer 1

Cleaning the data — then Isolation Forest scores every test

Data cleaning — what we do and why
⚑
Duration impossible (≤0 or >180 min) — set to null + flag duration_invalid=1. The flag is a fraud signal, not just a data error.
⚑
Odometer missing — keep null + flag odometer_missing=1. A tester who consistently omits odometer is behaving suspiciously.
⚑
Multiple fields missing simultaneously — add total_flags_count column. 5 missing fields = suspicious pattern, not carelessness.
✓
Date typos (e.g. 2094 → 2024) — correct where unambiguous, log correction for audit trail. Remaining invalid dates flagged.
✓
Time format errors (300am → 15:00) — standardise to 24hr. Out-of-hours tests (before 7am, after 8pm) flagged separately.
✓
Categorical inconsistencies — "Independent", "INDEPENDENT", "independent" all standardised to lowercase. Vehicle makes standardised via reference lookup.
↻
Duplicates — keep first occurrence, flag is_duplicate=1 on subsequent. Duplicate test submissions are themselves a signal.
✓
Feature scaling — StandardScaler applied. Ensures no single feature dominates the model due to larger numerical range.
Rule: never drop rows. Every anomaly in the data is a potential signal. Flags become features.
Isolation Forest — how it works
Reads one enriched row per test. Builds random decision trees. Measures how many splits it takes to isolate each test from all others. Fraudulent tests — unusual combination of short duration, high pass rate, rule fired — get isolated in very few splits. Normal tests take many splits.
Key: No fraud labels needed. Finds anomalies purely from statistical patterns. Works on the full unlabelled dataset — millions of tests.
Why not LOF or One-Class SVM? Isolation Forest scales linearly with data size, makes no distributional assumptions, and handles multiple features without degrading. LOF is too slow at DVSA scale. One-Class SVM requires careful tuning.
Contamination parameter: Set to 0.01–0.05 (1–5% expected fraud rate), consistent with published DVSA estimates of 0.5–2% irregular certificates.

Slide 4 of 5 — XGBoost + Composite Score + Enforcement Output

XGBoost adds supervision — then scores combine into one risk ranking

XGBoost — supervised layer
Trains on confirmed fraud cases (labelled set). Learns which feature combinations characterise confirmed fraud. Outputs fraud probability per test. Complements Isolation Forest — catches tests that resemble known fraud even if not statistically unusual.
Why XGBoost? Handles mixed feature types, robust to outliers, fast training, produces feature importance scores — you can show enforcement which features drove the flag.
Small labelled set? Use stratified k-fold cross-validation. Prevents overfitting. Validates model performs on unseen data despite small sample.
Confusion matrix produced here — true positives, false positives, false negatives tracked. False negatives weighted more heavily — missing fraud is worse than a wasted visit.
Composite risk score — three signals combined
composite = (0.50 × isolation_forest_score)
           + (0.30 × xgboost_fraud_probability)
           + (0.20 × domain_rule_fired)
Weights tuned using confirmed fraud cases. Reviewed every 3 months as feedback loop accumulates new labels.

Enforcement output — ranked list
Enforcement receives only HIGH and MEDIUM risk cases. Ranked by composite tester score. Each entry shows: tester ID, score, specific flagged test IDs, and plain English reason for each flag. LOW risk never sent — enforcement time is too valuable.
Tester ranking combines: count of flagged tests this month + severity of worst test + historical months flagged. A tester with 1 catastrophic test + 3 months history outranks a first-time tester with 5 moderate tests.

Anonymisation + Governance
The model works entirely on test IDs and tester IDs — anonymous identifiers. No names, addresses, or personal details in the analytical pipeline. Consistent with GDPR data minimisation and purpose limitation principles. Identifying details accessed only at point of enforcement investigation through DVSA's existing governed access process — consistent with how NHS patient IDs work in clinical systems. Lawful basis: DVSA's regulatory function to detect fraud in MOT testing.
Slide 5 of 5 — Validation · Trial · Future

Validation, controlled trial — and how the system keeps improving

Validation 1 — Development
Precision@K check — confirmed fraud cases must appear in top K anomaly scores. If not, features need improvement. Happens before deployment.
Confusion matrix (XGBoost) — true positives, false positives, false negatives tracked. False negatives weighted more heavily.
Cross-validation — stratified k-fold on labelled set. Prevents overfitting despite small confirmed fraud sample.
Score distribution check — confirmed fraud should cluster in top score percentile. Clear separation = model is working.
Validation 2 — Controlled Trial
Region A — uses the model. Enforcement guided by ranked risk list.
Region B — baseline. Traditional enforcement methods, no model.
Duration — 3 months live operation. Long enough for statistically meaningful comparison.
Metric — confirmed fraud found per enforcement inspection in each region. If Region A significantly higher → model validated for full rollout.
SME involvement — experienced enforcement officers in Region A review flags, provide verdicts, inform threshold tuning.
Expected result: Region A finds 40–80× more fraud per visit than Region B
Feedback Loop — Monthly Retraining
Enforcement verdict logged — confirmed fraud or false alarm recorded against test ID.
New label added — tester-month record gets confirmed fraud = 1. Enters training data.
Model retrains monthly — XGBoost gets stronger as labelled set grows. Weights retuned. Thresholds adjusted.
New fraud methods detected — Isolation Forest catches novel patterns. Feedback loop teaches XGBoost to recognise them next time.
Domain rules updateable — if DVSA changes threshold (e.g. 20 min → 30 min), rule updates immediately. No retraining needed for rule changes.
LLM Agent — Enforcement Intelligence Assistant
Natural language queries — officer asks "show all testers flagged 3+ consecutive months in South West." Agent queries database and responds in plain English.
Automated briefing drafts — when HIGH risk tester flagged, agent auto-drafts investigation report. Officer reviews, corrects, submits. Saves 30–60 min per case.
Anomaly explanations — officer asks "why is test 004 flagged HIGH?" Agent translates SHAP values into plain English: "duration 9 min is 94th percentile shortest for this vehicle class, combined with tester pass rate 16 points above peer average."
Human always decides — agent brings information. Officer makes the enforcement judgement. Not replaced — empowered.
Future Enhancements — Governance Required First
ANPR integration — confirms vehicle physically present at test site during test window. Catches ghost tests. Requires data sharing agreement with DVLA/National Highways.
CCTV + Convolutional Autoencoders — learns what legitimate MOT inspection looks like visually. Flags footage where tester barely inspects vehicle. Requires ICO guidance on automated video processing.
Network analysis — detects organised fraud rings where garages, mechanics, and testers work together. Requires graph database infrastructure.
All enhancements — require DPIA, lawful basis under UK GDPR Article 6, proportionality review. Privacy intrusion must be proportionate to fraud detection benefit.
"This system is designed to be deployable with what DVSA already has, explainable to enforcement officers and courts, and self-improving — so the longer it runs, the better it gets."
