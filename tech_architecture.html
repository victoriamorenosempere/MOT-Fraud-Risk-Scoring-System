Feedback Loop — step by step with software names
1
Enforcement officer enters verdict in case management system → AWS Lambda function fires automatically (event-driven, no manual trigger needed)
2
Lambda writes fraud_label = 1 back to the enriched table in AWS S3. Permanent record. This is called a write-back or label propagation.
3
On the 1st of each month, AWS EventBridge (scheduled rule — like a cloud calendar) triggers a SageMaker Pipeline. This retrains XGBoost on the updated labelled set. This process is called MLOps — Machine Learning Operations.
4
SageMaker evaluates new model vs old model on held-out test data. If Precision@K improves → new model version deployed automatically. Old version kept for rollback. Stored in SageMaker Model Registry.
5
Same EventBridge trigger also runs Isolation Forest on all new incoming test data. Scores written to S3. Power BI dashboard refreshes automatically via DirectQuery.
SQL — the difference between GROUP BY and Window Functions (PARTITION BY)
GROUP BY — collapses rows ✕ not what we want
SELECT tester_id,
  AVG(pass_rate) as avg_pass
FROM mot_tests
GROUP BY tester_id
Returns ONE row per tester. Loses all individual test detail. Cannot join back to test rows. Not useful for our enriched row.
PARTITION BY (Window Function) — keeps all rows ✓ what we use
SELECT test_id, tester_id, duration,
  AVG(pass_rate) OVER (
    PARTITION BY garage_type,
    region, volume_band
    ORDER BY test_date
    ROWS BETWEEN 30 PRECEDING
    AND CURRENT ROW
  ) as peer_pass_rate
FROM mot_tests
Returns ALL rows. Attaches the rolling 30-day peer average to every individual test row. The row stays intact. This is what feeds Isolation Forest.
Why PySpark from day one
Even the initial historical dataset will be large — years of MOT records across 32M tests per year. Pandas loads everything into memory on one machine. At this scale it will crash or be extremely slow. PySpark (via AWS Glue or AWS EMR) distributes the data across multiple machines and processes it in parallel. We use PySpark for the SQL joins and feature engineering, then pass the enriched feature matrix (much smaller — only the relevant columns) to SageMaker where Pandas and scikit-learn handle model training fine.
The data journey in code terms
AWS Glue (PySpark) — reads raw S3 parquet, runs SQL window functions, writes enriched feature table back to S3

AWS SageMaker (Python/Pandas) — reads enriched feature table, runs Isolation Forest + XGBoost, writes scores back to S3

AWS Athena (SQL) — queries scores directly on S3 without loading into a database. Power BI connects here.
Quick-fire Q&A — know these cold
What is S3?
Amazon Simple Storage Service. Cloud file storage — like a hard drive in the cloud. Stores our raw data tables and model files as Parquet or JSON files.
What is AWS Glue?
Amazon's managed ETL service. Runs PySpark jobs in the cloud without needing to set up servers. We use it for SQL joins, window functions, and writing enriched data back to S3.
What is SageMaker?
Amazon's managed machine learning platform. Gives you Jupyter notebooks in the cloud, model training, model registry, and deployment pipelines. No laptop needed.
What is AWS Lambda?
A function that runs automatically when triggered by an event. We use it to write fraud verdicts back to S3 the moment an enforcement officer enters them — no scheduled batch needed.
What is AWS EventBridge?
A cloud scheduler. Like setting a calendar reminder that triggers code automatically. We use it to trigger monthly model retraining on the 1st of each month.
What is Parquet?
A columnar file format — much faster to query than CSV because you only read the columns you need. Standard format for big data on AWS. PySpark and Athena both read it natively.
What is Athena?
AWS service that lets you run SQL queries directly on files in S3 — no database needed. Power BI connects to Athena to show live scores without importing data.
What is MLOps?
Machine Learning Operations — the practice of automating model training, evaluation, versioning, and deployment. Means the model retrains and redeploys itself monthly without manual intervention.
Why Parquet not CSV?
Parquet is columnar — reading 5 columns from 32M rows reads only those 5 columns from disk. CSV reads every column. 10-100x faster for our analytics queries. Also compresses much better.
Could this run on Azure instead of AWS?
Yes — Azure Data Factory replaces Glue, Azure Databricks replaces SageMaker/PySpark, Azure Blob Storage replaces S3, Azure ML replaces SageMaker. Same architecture, different vendor names. DVSA uses whatever Gov UK cloud framework specifies.
Why not just keep it in a SQL database?
A traditional SQL database stores data on one server — 32M rows per year becomes billions over time and query performance degrades. S3 + Athena scales infinitely and costs near zero for storage. Compute only costs when you query.
What is a time-based train/test split?
We train XGBoost on fraud cases confirmed before a cutoff date — say Jan 2024. We test it on cases after that date. This prevents data leakage — the model never sees future outcomes during training, mimicking real deployment conditions.
